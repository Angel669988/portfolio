Memory 系统 · 知识结构图

Agent 架构纵向贯穿线 · 主要作用于 L1→L3 输入端 · 让产品有"熟人感"
定位提醒：以下是体系认知。真实情况——我的产品用对话系统通用 memory 能力，只有单会话工作记忆，跨会话未做。懂得深是为了判断准，不等于"做过 memory 系统"。
01 分类体系 · 两种分法
按时间跨度分（记多久）
短期 / 工作记忆
单会话 context 内，对话结束就没。我的产品在这一层。（= 状态对象/多轮管理）
长期记忆
跨对话持久化，存外部、用时检索。真正难的部分，多数团队没做好。
用户档案
跨会话的结构化稳定认知——他是谁、习惯什么、历史行为。
按内容分（记什么）· 认知科学启发，面试显专业
情景记忆
"发生过什么"——具体事件带时间。= 日记
语义记忆
"关于用户的稳定事实"——从情景提炼的结论。= 从日记总结的关于你的事实
程序记忆
"学会怎么更好服务"——个性化做事方式。最高级，多数产品没做到
提炼链：情景 → 语义（从"发生了什么"总结出"这意味着什么"）是 memory 核心动作之一。

03 跨会话这条线为什么难跨 · 四道坎 短期记忆简单（就在 context 里）· 跨会话难在这四件事
坎 1
抽取
不能存整段对话，要用 LLM 从对话提炼该记的（情景→语义）。抽错就记错。
坎 2
存储
结构化（精确但只能存预设字段）vs 向量化（灵活但检索易污染）。成熟做法两者混用。
坎 3 · 最核心
检索相关性
调太多→Lost in the Middle 淹没当前；调错→乱记；调出过时记忆自信用错→比不记更糟。"记住不难，难在对的时候想起对的事"
坎 4
更新与遗忘
记忆会过期，新旧冲突难自动检测，记忆库会越来越脏。
关键产品洞察：产品不做跨会话往往是安全的默认——检索没做好时，记错比不记更伤体验。"判断当前做不好不如不做"本身就是一个产品观点。
04 三个核心难题 · 技术细节
检索相关性
一路打补丁：
① 语义相似度（相似≠相关）
② +时间衰减 recency
③ +重要性加权 importance
④ 查提炼后的洞察，非原始流水账
冲突消解
多数系统最弱：
① 难自动发现冲突
② 信谁——默认信新的，但要区分"永久偏好改变" vs "一次性临时需求"
业界无好的通用解
短期→长期提升
实时提升（不遗漏但贵）
vs 会话后批量提升（用 LLM 通读总结，有全局视角，多数成熟做法用这个）
面试金句："记住"和"检索"大家都做，"更新和冲突消解"是分水岭——要判断用户是真变了还是这次特殊，目前没有好的自动化方案。
引用资本：斯坦福 Generative Agents 论文把检索系统化为"相关性 + 时间新近 + 重要性"三因子加权。 ｜ 连接我的产品："单会话记得、之后忘" = 缺了"会话结束时把工作记忆提炼并持久化"这一步。
05 应用层判断 + 诚实话术 复用 RAG 心智：底层不是我做的，但怎么用好是我的活
① 什么该记
业务决策。行业/文档风格偏好该记，一次性内容不该记
② 何时调用
应用层决策。简单任务不调，个性化才调，避免污染上下文
③ 用错代价
决定要不要用。没把握做好前，记错比不记更伤体验
④ 体验底线
记错能否纠正？用户能否看到/删除被记内容（隐私合规）
最值钱的一句：把"没做跨会话"从事实，转化为"懂克制、懂权衡"的产品判断——"正因为我理解它多难，才判断暂时不做。" 认知是为判断服务，不是为证明做过。
